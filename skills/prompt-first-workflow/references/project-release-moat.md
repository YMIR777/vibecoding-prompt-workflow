# Project Release Moat

Before any public repository, release, package, model, dataset, document, skill, template, or creative-work publication, expose and confirm these choices:

1. **Authorship**: author, first publication date, sources, contributors, and version history.
2. **Canonical version**: the one official repository or release and how users identify it.
3. **Brand narrative**: project name, searchable positioning, principles, and differentiation.
4. **Openness boundary**: visibility, permitted users, commercial use, private modification, public derivatives, redistribution, and written-permission requirements.
5. **Asset types**: code, text/methodology, design, data, model, and brand may require different licenses.
6. **Contribution model**: issues, pull requests, CLA/DCO, or no external contributions.
7. **License**: explain trade-offs and obtain the user's explicit choice. Never infer MIT from “public” or “open.”
8. **Irreversibility test**: would the user still accept the license if a fork became more popular, a third party charged for it, or attribution became unclear?
9. **Release evidence**: secrets removed; documentation, examples, installation, tests, version, attribution, and license verified.

Use this prompt fragment:

```text
Release purpose:
Authorship and provenance:
Canonical repository / release:
Brand and positioning:
Permitted use and users:
Commercial use:
Modification, derivatives, and redistribution:
License:
Contribution model:
Irreversibility test:
Release verification:
```

Do not create a remote repository, push, publish, or release until all material rights and external-write choices are confirmed.
