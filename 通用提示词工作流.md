# 通用提示词工作流

适用对象：任何聊天式大模型、Coding Agent、研究 Agent 或工作流 Agent。

## 一句话原则

不要把提示词当咒语。把模糊意图翻译成一个有边界、能执行、可验收、可验证的任务；由证据而不是感觉决定是否完成。每一次任务都要先展示这版可执行提示词，既提高执行质量，也让使用者持续练习任务定义。

## 第一性原理检查

“从第一性原理思考”不是一句让模型变聪明的口号，而是要求它回到任务不可省略的基础：

1. 区分已知事实、未验证假设和主观偏好。
2. 明确最终结果与不可违反的约束。
3. 定义可观察的成功标准和验证证据。
4. 展示会改变结果的关键选择，由用户决定。
5. 在这些基础明确后，选择最小可验证路径。

## 每次任务的六步

1. **结果**：先说用户最终得到什么，而不是先指定技术方案。
2. **上下文**：给与当前决策有关的文件、现状、数据、例子、错误、约定和已尝试路径。
3. **边界**：写清范围、不可改变项、非目标、成本/时间/性能/安全/风格约束，以及哪些动作必须先确认。
4. **验收**：把“好、简单、智能、漂亮、完成”换成可以观察或测试的条件。
5. **小步执行**：先读和计划，再做一个能独立验证的小切片；失败先找根因。
6. **证据交付**：说明改动/结论、实际验证、未测风险和最小下一步。

## 给任意 Agent 的通用系统提示词

```text
对我的每一个任务，先将原始请求整理为简洁、可执行、可验证的任务提示词，并在开始工作前可见地展示它。即使是简单任务，也至少展示一行简短版本。

先识别：目标与用户场景、相关上下文、范围与约束、不可改变项、成功标准与验证证据、需要的交付格式，以及可自主执行与必须确认的动作。

只补充会改变决策的信息；不要堆砌背景，不要重复指令。将主观要求转为可观察标准。若未定项会实质改变交付物、范围、安全、架构、验收、集成、发布或成本，先列出关键选择、说明每项影响，并等待我确认；不要静默代替我选择。否则说明可逆的低风险假设并继续。

对于变更、构建或修复：先阅读现状并给出最小可行计划；按可独立验证的小步骤实施；运行实际验证。对于回答、解释、审查、诊断或计划：只检查和报告，除非我明确要求修改。

除非我明确说“直接执行”“跳过提示词整理”或“不触发提示词工作流”，否则必须先展示一版简短的“可执行提示词”。完成时报告：结果/改动、验证证据、未测风险或不确定性、最小下一步。不要把“应该可以”当作完成。
```

## 最小任务模板

```text
目标/结果：
使用者与场景：
相关上下文：
范围、约束与不可改变项：
成功标准/验收证据：
需要的交付格式：
可自主执行与必须确认的动作：
```

不需要每次都填满。删除对决策没有影响的字段，绝不虚构信息。

## 必须确认的关键选择

以下情况不能静默采用默认值，必须先向用户列出选择并确认：答案会改变交付物、范围、安全、架构、验收、集成、发布、成本或维护责任。

例如，创建可复用 Skill 或文档时，至少检查：

- 交付物是 Skill、通用文档，还是两者都要？
- 哪些 Agent 和纯聊天模型需要支持？
- 自动触发、手动触发，还是每次可见展示？
- 只做提示词整理，还是扩展到计划、执行、验证和汇报？
- 英文、中文，还是双语？
- 是否需要安装说明、可复制系统提示词、案例、评估、许可证、GitHub 发布或插件封装？

不要为了显得仔细而追问无关细节。能从现有材料安全发现的内容应自行查找；只有会改变结果的选择才提问。

## 任务模板

### 新项目

```text
目标：为 [谁] 做 [什么产品]，使其能 [核心任务]。
核心流程：[开始] -> [关键操作] -> [成功结果]。
首版范围：包含 [功能]；不包含 [明确非目标]。
上下文：[现有项目、参考、数据、品牌/技术约定]。
约束：[平台、预算、隐私、性能、兼容性、设计]。
验收：[可观察行为和验证方式]。
交付：[代码/方案/原型/文档]，格式为 [格式]。
权限：可直接做 [安全本地动作]；做 [外部写入/发布/破坏性动作] 前先确认。
```

### 修改或开发

```text
目标：将 [当前行为] 改为 [期望行为]。
上下文：[相关文件、现有行为、示例、日志]。
约束：保持 [兼容性或不变量]；不要改 [范围外区域]。
验收：[测试、UI 状态、API 行为、性能阈值]。
请先阅读现状，提出最小可行方案，实施并运行验证，报告证据和剩余风险。
```

### 排错

```text
问题：[完整错误或不正确行为]。
复现：[步骤、输入、环境]。
预期/实际：[对比]。
上下文：[日志、代码、近期改动]。
先基于证据验证主要假设；确认根因后再修复；重新运行原始复现路径。
```

### 研究或解释

```text
问题/决策：[需要知道或决定什么]。
受众与深度：[谁使用、需要多深]。
范围：[包含/排除、地区、时间]。
证据：[要用的来源与时效要求]；区分事实、解释和不确定性。
输出：[结论/比较/建议/报告]，格式为 [格式]。
```

## 反模式

- “帮我做个好看的网站”，却没有用户、内容、范围或验收标准。
- 把登录、支付、后台、部署、设计等多个大目标塞进一个不可验证的请求。
- 只背模板，却不给真实代码、数据、日志或限制。
- 跳过构建、测试、浏览器检查或来源核验，只接受“应该没问题”。
- 为一个小问题让 Agent 重写架构。

## 交付前五问

1. 改了什么或得出了什么结论，为什么？
2. 它影响什么用户行为或决策？
3. 跑了什么真实验证，结果是什么？
4. 还有什么未测风险、不确定性或范围外内容？
5. 最小的下一步是什么？

## 来源与边界

本工作流综合：

- IFQ.AI 的 Vibe Coding：意图、上下文、切片、验证、迭代，以及“AI 写代码；证据决定完成”。
- OpenAI GPT-5.6 模型指南：目标导向提示词、相关而精简的上下文、明确权限边界、具体输出与语气要求、按代表性任务验证。

它是提高任务定义和验证质量的工作流，不保证模型永远正确。当前事实、API 行为、法律/医疗/金融建议与外部数据仍需按任务重新验证。
