# 项目发布与创作护城河

项目发布不是简单上传文件，而是在决定谁能看、使用、修改、传播和商业化。发布前必须先保护作品归属和未来选择权，再决定开放程度。

## 四层护城河

### 作者身份

- 在 README 中注明作者、首次发布日期、来源和贡献者。
- 用 Git 历史、版本标签和变更记录保存创作证据。
- 区分原创、引用与第三方内容。

### 官方版本

- 指定唯一 canonical repository 和 official release。
- 让 README、主页和发行说明始终指向官方版本。
- 用持续维护、验证记录和兼容性建立信任。

### 品牌与叙事

- 使用清晰、可搜索、能长期延续的名称与一句话定位。
- 说明解决的问题、核心原则和与相似方案的差异。
- 保留名称、Logo 和“官方版本”称谓的管理权；影响力增长后再评估商标。

### 有限开放与升级权

- 根据目的选择私有、受限公开、非商业、禁止衍生、copyleft、宽松开源或双许可证。
- 先保留、以后放开通常可行；宽松授权一旦发布，通常不能对既有接收者收回。
- 用真实使用和贡献证据决定是否进一步开放，而不是为了“开源”标签提前授权。

## 发布前必须确认

```text
发布目的：展示、教学、协作、采用、获客、商业化或作者影响力？
作者身份：作者、日期、来源、贡献者和版本历史是否清楚？
官方版本：唯一官方仓库或发行地址是什么？
品牌叙事：名称、定位、原则和差异是什么？
资产类型：代码、文本、设计、数据、模型和品牌分别如何授权？
开放边界：谁能用，能否商业使用、修改、Fork、再发布或制作衍生品？
许可证：用户是否理解并明确选择？
贡献机制：是否接收 Issue、PR、CLA/DCO？
不可逆测试：如果 Fork 更火、第三方收费或署名模糊，是否仍接受当前授权？
发布证据：密钥、隐私、文档、安装、测试、版本和许可证是否已检查？
```

关键权利没有确认前，不创建远程仓库、不推送、不发布版本。

## 许可证不是唯一护城河

许可证提供规则和维权基础，但不能阻止所有模仿。长期护城河来自：清晰作者身份、可信官方版本、品牌叙事、持续更新、验证质量和用户关系。

宽松开源适合需要生态采用和外部贡献的项目；受限公开适合重视创作控制、方法论归属和未来商业授权的项目。选择应由发布目标决定，而不是由“公开”这个词自动决定。

## 发布后维护

- 发布版本、变更记录、验证证据和兼容性说明。
- 维护官方链接、作者信息、引用格式和搜索可发现性。
- 收集真实使用、反馈和贡献，再评估授权范围。
- 定期检查依赖许可证、品牌混淆和未经授权的再发布。
